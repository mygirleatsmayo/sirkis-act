## Problem Statement

Three systemic issues prevent the theme system from scaling to visually diverse themes:

1. **HSL derivation is perceptually non-uniform.** `shiftLightness` in `derivationRules.ts` uses HSL, where a +5 L shift on yellow produces a vastly different contrast step than +5 on blue. This forced `textSecondary` and `textSubtle` to be demoted to standalone primaries because HSL derivation couldn't produce consistent results across hues.

2. **Hardcoded Tailwind opacities assume a dark-mode-first world.** `bg-white/10`, `bg-black/20`, `border-white/15`, and similar classes appear 34 times across `App.tsx` (16), `SettingsModal.tsx` (14), and `Root.tsx` (4). These break on the light `Overheated Rhizome` theme: white overlays are invisible on warm cream backgrounds, and white borders vanish.

3. **Inline `rgba` in styles is mode-biased.** Debossed text shadows (`rgba(255,255,255,0.12)` highlight + `rgba(0,0,0,0.7)` shadow), swatch rings (`rgba(255,255,255,0.15)`), and the sidebar inset glow all assume dark surfaces.

**Success criteria:** After implementation, (a) all `shiftLightness` calls are replaced with OKLCH equivalents, (b) zero `bg-white/N`, `bg-black/N`, `border-white/N`, `border-black/N`, or `text-white/N` classes remain in `App.tsx`, `SettingsModal.tsx`, or `Root.tsx`, (c) `textSecondary` and `textSubtle` can be optionally OKLCH-derived from `bg`, and (d) both Cyprus and Overheated Rhizome render correctly without manual per-theme opacity tuning.

## Color Quality Principles

These principles from the `/colorize` framework govern decisions throughout this plan. We are not changing palettes or re-theming; we are ensuring the derivation and token system follows best design practices.

- **Avoid pure black/white in overlays where practical.** When deriving surfaces, borders, and overlays, prefer the theme's own `textPrimary` as the overlay base rather than hardcoded `rgba(0,0,0,...)` or `rgba(255,255,255,...)`. On themes with tinted primaries (Rhizome: `#180C08`), this naturally produces warm, hue-coherent overlays. On themes with pure white text (Cyprus: `#ffffff`), the composited result is still tinted by the surface beneath at low alpha, so the visual difference is negligible.
- **Pure white text on Cyprus is a conscious brand choice**, not a deficiency. Theme authors can always use off-whites (e.g., `#f0f4f8`) for softer contrast. The derivation system does not force this.
- **`rgba(0,0,0,alpha)` at low alpha (≤ 0.08) is acceptable** for universally darkened surfaces like `mutedBg` and `surfaceSunken`, because at such low opacity the composited result inherits the hue of whatever surface lies beneath. No perceptible gray cast.
- **OKLCH for all lightness operations.** Because OKLCH L is perceptually linear, a +0.05 shift on teal looks the same as +0.05 on gold. This eliminates the hue-dependent inconsistency that forced `textSecondary`/`textSubtle` into standalone primaries under HSL.

## Phase 1: OKLCH Core Utilities

**File:** `src/utils/colorMath.ts`

Add pure-math OKLCH conversion functions with zero external dependencies. The conversion chain is sRGB → Linear RGB → CIE XYZ (D65) → OKLab → OKLCH (and inverse).

### New Functions

```ts
type Oklch = [number, number, number]; // [L: 0–1, C: 0–~0.4, H: 0–360]

/** Convert hex (#RRGGBB) to OKLCH [L, C, H] */
export const hexToOklch = (hex: string): Oklch

/** Convert OKLCH [L, C, H] to hex (#rrggbb), clamping to sRGB gamut */
export const oklchToHex = (L: number, C: number, H: number): string

/** Shift OKLCH lightness by deltaL (clamped 0–1). Preserves chroma and hue. */
export const shiftOklchLightness = (hex: string, deltaL: number): string

/**
 * Derive a text color from a background hex at a target OKLCH lightness.
 * Inherits hue from bgHex, clamps chroma to keep text readable.
 * Use for perceptually uniform secondary/subtle text derivation.
 */
export const deriveTextFromBg = (
  bgHex: string,
  targetL: number,
  maxChroma?: number, // default 0.035
): string
```

### Conversion Pipeline Detail

**Gamut clamping:** `oklchToHex` must clamp each sRGB channel to [0, 255] after the inverse pipeline. Highly chromatic OKLCH values can produce out-of-gamut RGB. Simple per-channel clamping is sufficient for this use case (we are shifting lightness by small deltas on already-in-gamut theme colors, so out-of-gamut results are rare and minor). No sophisticated gamut mapping (e.g., CSS `color()` with `in srgb`) is needed.

The implementing agent should use these matrix coefficients:

**Linear RGB → XYZ (D65):**
```
X = 0.4122214708·R + 0.5363325363·G + 0.0514459929·B
Y = 0.2119034982·R + 0.6806995451·G + 0.1073969566·B
Z = 0.0883024619·R + 0.2024326343·G + 0.9505321522·B
```

**XYZ → OKLab (via LMS cone response):**
```
l = cbrt(0.8189330101·X + 0.3618667424·Y − 0.1288597137·Z)
m = cbrt(0.0329845436·X + 0.9293118715·Y + 0.0361456387·Z)
s = cbrt(0.0482003018·X + 0.2643662691·Y + 0.6338517070·Z)

L = 0.2104542553·l + 0.7936177850·m − 0.0040720468·s
a = 1.9779984951·l − 2.4285922050·m + 0.4505937099·s
b = 0.0259040371·l + 0.7827717662·m − 0.8086757660·s
```

**OKLab → OKLCH:**
```
C = sqrt(a² + b²)
H = atan2(b, a) × 180/π   (normalize to 0–360)
```

Inverse uses the transposed/inverted matrices. `oklchToHex` must clamp the resulting sRGB channels to [0, 255] to stay in gamut. For `deriveTextFromBg`, the function sets L to the target value, clamps C to `maxChroma`, and preserves H from the background, producing text that is perceptually harmonious with the background hue.

### sRGB Gamma

Use the same linearization already in `relativeLuminance`:
```
sRGB ≤ 0.04045 → linear = sRGB / 12.92
sRGB > 0.04045 → linear = ((sRGB + 0.055) / 1.055) ^ 2.4
```
Inverse (linear → sRGB):
```
linear ≤ 0.0031308 → sRGB = linear × 12.92
linear > 0.0031308 → sRGB = 1.055 × linear^(1/2.4) − 0.055
```

## Phase 2: Replace HSL Derivations with OKLCH

**File:** `src/themes/derivationRules.ts`

### 2a. Replace `shiftLightness` Calls

Every `shiftLightness(hex, delta)` call uses HSL L (0–100 scale). Replace with `shiftOklchLightness(hex, deltaL)` where deltaL is on the OKLCH 0–1 scale. The perceptual equivalents (approximate, tune visually):

| Current HSL Call | OKLCH Replacement | Notes |
|---|---|---|
| `shiftLightness(p.bg, 2 * dir)` → bgGlass | `shiftOklchLightness(p.bg, 0.02 * dir)` | Subtle elevation |
| `shiftLightness(p.bg, -3 * dir)` → bgInput | `shiftOklchLightness(p.bg, -0.03 * dir)` | Sunken input |
| `shiftLightness(p.bg, 8 * dir)` → borderDefault | `shiftOklchLightness(p.bg, 0.08 * dir)` | Visible border |
| `shiftLightness(p.brand, -5 * dir)` → sliderAccentHover | `shiftOklchLightness(p.brand, -0.05 * dir)` | Hover darken |
| `shiftLightness(p.bg, -30)` → bgOverlay base | `shiftOklchLightness(p.bg, -0.30)` | Overlay darken |

The delta values are intentionally 1:100 mapped (HSL delta 2 → OKLCH delta 0.02) because OKLCH L runs 0–1 and the shifts are small. Visually verify on both themes and adjust if needed.

### 2b. Replace `toggleOff` HSL Derivation

Current logic:
```ts
const [bgH, bgS] = hexToHsl(p.bg);
return hslToHex(bgH, Math.min(bgS * 0.2, 25), 73 + 10 * dir);
```

OKLCH replacement:
```ts
const [bgL, bgC, bgH] = hexToOklch(p.bg);
const targetL = mode === 'dark' ? 0.75 : 0.65;
return oklchToHex(targetL, Math.min(bgC * 0.15, 0.03), bgH);
```

This preserves the intent: a desaturated, mid-lightness track color with the background's hue, adjusting lightness by mode.

### 2c. Fix `borderSubtle` Derivation

Currently `borderSubtle` uses hardcoded pure white/black with a luminance branch:

```ts
borderSubtle: relativeLuminance(p.bg) < 0.5
  ? 'rgba(255, 255, 255, 0.06)'
  : 'rgba(0, 0, 0, 0.06)',
```

Replace with text-relative derivation (same approach as the new `borderMuted`):

```ts
borderSubtle: hexToRgba(p.textPrimary, 0.06),
```

This is mode-adaptive without branching. On Cyprus (textPrimary `#ffffff`): `rgba(255,255,255,0.06)` — identical to current. On Rhizome (textPrimary `#180C08`): `rgba(24,12,8,0.06)` — identical to the explicit value already authored in that theme file. The `relativeLuminance` import in `derivationRules.ts` can be removed after this change (it is only used here and in `ThemeLab.tsx` which imports it separately).

### 2d. Update Import

```ts
// Before

// After
import { shiftOklchLightness, hexToOklch, oklchToHex, hexToRgba } from '../utils/colorMath';
```

Remove `hexToHsl`, `hslToHex`, `shiftLightness`, and `relativeLuminance` from this file's imports. They remain exported from `colorMath.ts` for other consumers.

### 2e. Derivable `textSecondary` and `textSubtle` (Opt-in)

Add commented-out derivation formulas to `applyDerivations` so they can be activated per-theme or globally in a future step. The current pass-through (`textSecondary: p.textSecondary`) remains the default.

```ts
// Future OKLCH derivation (activate when themes opt in):
// textSecondary: deriveTextFromBg(p.bg, mode === 'dark' ? 0.82 : 0.30, 0.03),
// textSubtle:    deriveTextFromBg(p.bg, mode === 'dark' ? 0.55 : 0.50, 0.02),
textSecondary: p.textSecondary,
textSubtle: p.textSubtle,
```

The commented formulas use `deriveTextFromBg`:
- **textSecondary dark**: L=0.82 (bright but not white), C≤0.03 (slight hue tint), H=bg hue
- **textSecondary light**: L=0.30 (dark but not black), C≤0.03
- **textSubtle dark**: L=0.55 (muted gray), C≤0.02
- **textSubtle light**: L=0.50 (muted), C≤0.02

To activate: replace the pass-through with the derivation call. Themes that set explicit values in their `.ts` file will still override via the existing primary-protection logic.

### 4e. Debossed `textShadow` Patterns

Four instances in `App.tsx` (lines 953, 958, 1023, 1061) use hardcoded debossed text shadows:

```
textShadow: '0px 1px 0px rgba(255,255,255,0.12), 0px -1px 0px rgba(0,0,0,0.7)'
```

These have two components:
- **Top highlight**: `rgba(255,255,255,0.12)` — simulates light from above hitting a raised surface
- **Bottom shadow**: `rgba(0,0,0,0.7)` — simulates shadow beneath

On a light theme, the highlight should be stronger white and the shadow should be lighter.

**Solution:** Add deboss properties to `ThemeEffects`:

```ts
// In ThemeEffects interface (src/themes/types.ts):
deboss?: {
  highlight: string;   // top-edge highlight rgba
  shadow: string;      // bottom-edge shadow rgba
};
```

**Cyprus values:**
```ts
deboss: {
  highlight: 'rgba(255, 255, 255, 0.12)',
  shadow: 'rgba(0, 0, 0, 0.7)',
},
```

**Overheated Rhizome values:**
```ts
deboss: {
  highlight: 'rgba(255, 255, 255, 0.25)',
  shadow: 'rgba(0, 0, 0, 0.15)',
},
```

**CSS var sync** (add to `syncCssVars.ts`):
```ts
if (theme.effects.deboss) {
  root.style.setProperty('--deboss-highlight', theme.effects.deboss.highlight);
  root.style.setProperty('--deboss-shadow', theme.effects.deboss.shadow);
}
```

**Consumption** (replace inline styles in `App.tsx`):
```ts
textShadow: `0px 1px 0px var(--deboss-highlight, transparent), 0px -1px 0px var(--deboss-shadow, transparent)`
```

The CSS `var()` fallback to `transparent` means themes without `deboss` defined get no text shadow, which is a safe default.

## Phase 6: Theme File Updates

After derivation rules are updated, the exact hex/rgba outputs for derived tokens will shift slightly due to OKLCH vs HSL differences. The theme `.ts` files must be updated to match the new derivation output so they remain the source of truth.

**Files to update:** `cyprus.ts`, `overheatedRhizomeLight.ts`

For each theme:
1. Run `applyDerivations(extractPrimaries(theme.colors), mode)` with the new OKLCH engine
2. Compare output to current theme file values
3. Update any derived token values that changed (e.g., `bgGlass`, `bgInput`, `borderDefault`, `sliderAccentHover`, `bgOverlay`, `toggleOff`)
4. Add the three new tokens (`surfaceHover`, `surfaceSunken`, `borderMuted`)
5. Add `deboss` to effects

**Do not change primary token values.** Only derived tokens should shift. If a derived value shifts more than a few perceptual units, investigate and adjust the OKLCH delta.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Visual drift on existing themes after HSL → OKLCH | OKLCH delta values are 1:100 mapped from HSL. Verify each derived token against the current theme file value; adjust deltas if the shift exceeds ±2 hex digits per channel. |
| Out-of-gamut OKLCH results | Per-channel sRGB clamping. Only affects extreme chroma cases, which theme primaries don't hit. |
| Token contract bloat | Only 3 new `ThemeColors` tokens + 1 optional `ThemeEffects.deboss`. Each maps to multiple repeated hardcoded patterns; none are one-offs. |
| Theme Lab regression during theme switch | Existing `initThemeIdRef` guard and `playground` filter are unaffected by this plan. Theme Lab is out of scope for opacity migration. |
| Runtime derivation leaking into theme load/switch | No change to the derivation boundary. `applyDerivations` is only called by `applyLockedDerivations` in Theme Lab. `ThemeProvider` loads from `getTheme()` → `resolveTheme()` with zero derivation, unchanged. |
